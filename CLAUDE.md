## Build / run

- **Imports are EXTENSIONLESS and inlined** (`#import [ "libxmp" cdecl [...] ]`, `"SDL3"`) — no single-use `#define` macros for library names (user rule). The toolchain decides the link mode at the command line: **shared (.dll) by default**, **static (.lib archives) with `--static`**.
- Main source is **`player.red`**; the dev binary `player.exe` stays LOCAL (gitignored), the published release binary is **`CherryTracker.exe`** (in the repo — produced with `-o`).
- Dev build (dynamic, console subsystem): `D:\DevAI\Tools\redc.exe -c player.red` → `player.exe` — needs `libxmp.dll` + `SDL3.dll` + `libRedRT.dll` alongside. Extensionless PE import names resolve because the Windows loader appends `.dll`.
- **Release build (static, self-contained, GUI subsystem) — the verified command (user's up-to-date `Tools\redc.exe`; mine may lack `-s`):**
  ```
  ..\tools\redc.exe -r -s -t Windows -o CherryTracker.exe player.red
  ```
  `-s` switches the extensionless imports to the staged static archives; `-t Windows` drops the console for the shipping exe (dev builds keep it).
- Static archives staged in the project: **libxmp.lib** (clang-cl x86: `cmake -S libxmp -B build-static -A Win32 -T ClangCL -DBUILD_STATIC=ON` → target `xmp_static` → copy `build-static\Release\libxmp-static.lib` to `libxmp.lib`) and **SDL3.lib** (from `..\sdl-red\`).
- Headless smoke test: `redc -r -o test-play test-play.reds` then `test-play.exe` (plays ~8 s, prints pushed/queued).

## Architecture

- `xmp.reds` — libxmp cdecl `#import` + offset-based readers for `xmp_frame_info` / `xmp_module`. R/S `struct!` can't express the embedded `channel_info[64]` array, so fields are peeked by byte offset (`peek-i32`/`peek-u8`); offsets verified vs `include/xmp.h` @ 4.7.0 on x86. Modules load via `xmp_load_module_from_memory` (Red `read/binary` → `binary/rs-head`), avoiding path-encoding issues.
- `audio.reds` — minimal SDL3 (`SDL_Init` / `SDL_OpenAudioDeviceStream` / `Put` / `GetQueued`), S16 / 2ch / 44100, manual feed. SDL runs its own audio thread; we just keep its queue topped up.
- `player.red` — `#system-global` includes + player state + R/S helper funcs (pump / sync / ring) + `routine!` bridges (`pt-*`), then the Red Draw renderer + View loop.
- **The render path is ZERO-ALLOCATION (do not regress it).** Baseline cost of the naive draw-block-per-frame style was **10 GC runs/sec** (~23 MB/s of garbage, measured with a `-r -d` build — debug runtime prints one `runs: N … mark/sweep ms` line per collection); after the rework it is **0 GC runs over 30 s of playback**. The rules:
  - ONE persistent `frame-blk` is installed as `canvas/draw` ONCE; `render` does `clear frame-tail` (keeps the `anti-alias`/`scale` prefix) and re-appends every layer. `clear` keeps capacity → steady-state appends never allocate. Repaint is driven by the ownership events of the mutations themselves (no `show`, no per-frame `canvas/draw:` assignment).
  - Every helper is an `emit-*` that appends into `out` via **`compose/into [...] tail out`** — no intermediate blocks, parens produce immediates (pair!/tuple!/integer!). Never return fresh blocks / `make block!` / plain `compose` in the frame path.
  - **No string is ever built during a frame.** Static text comes from tables built once at startup (`note-strs`, `hex-strs`, `row-strs`, `chw-strs`/`chn-strs`, `state-tags`); live values go into dedicated per-slot reused buffers (`pos-buf`, `el-buf`, …) via `clear` + `pad2-into`/`pad3-into`/`fmt-time-into` (`append buf <int>` forms digits without garbage). Each draw slot needs its OWN buffer (the frame block holds references). Pattern cells use the grow-on-demand `cell-pool`. Load-time-only strings (`name-cache`, `file-line`, truncations) are prepared in `load-file`, never per frame.
  - `measure-*` / `text-size` (face-facet writes) are first-tick only — `emit-btn-glyph` reads the cached `icon-szs`.
  - No manual `recycle` anywhere — with clean frames it only adds pauses.
  - Per-frame `foreach` is avoided in emitters (allocates its context each call) — indexed `while` walks instead.
  - **Verifying render refactors:** dump `mold canvas/draw` at a deterministic tick (idle frame + a forced loaded/paused frame with seeded `vu-level`/`spec-level`/`spec-peak`, forced `view-pat`/`view-row`, a pressed button) from the old and new builds and byte-diff — proves the emitted Draw stream identical without pixel comparison.
- **Sync:** a snapshot-FIFO ring stores, per rendered frame, `{end-bytes, pos, pattern, row, bpm, …, per-channel vol/note, decimated scope}`. `pt-sync` surfaces the snapshot whose `end-bytes ≤ (total_pushed − SDL_GetAudioStreamQueued)` = the currently-audible frame → visuals lock to the sound regardless of how far audio is buffered ahead.

## Hard-won gotchas (each cost a build cycle — see linked memories)

- **SDL3 `bool` returns marshal unreliably through R/S** (only AL set, upper EAX garbage). Don't gate on `SDL_Init`'s return — gate on `SDL_GetCurrentAudioDriver` being non-null. (memory `feedback_red_ffi_bool_return`)
- **DPI + resizable letterbox layout:** Red/View is per-monitor-aware via its manifest — do **not** call `SetProcessDPIAware` yourself (it conflicts and breaks coords). The window is **resizable + maximizable** (`win/flags: [resize]` → `WS_THICKFRAME` + keeps the maximize box, per `gui.reds` OS-make-view). The 1024×768 design renders through a **letterbox transform rebuilt on every resize** (`rebuild-prefix`): `fit-s = min(cw/1024, ch/768)` (largest 4:3 fit), centred via `translate fit-ofs`, with a full-canvas `col-bg` box behind it so the margins read as chrome gray, not black bars. This subsumes the old fixed DPI scale — at the initial size `fit-s == draw-scale == 1/dpi` and `fit-ofs == 0`, so the default look is unchanged. The window's `on-resize` actor does `canvas/size: face/size` + `rebuild-prefix face/size` + `render`; setting `canvas/size` there does **NOT** feedback-loop (the window doesn't auto-resize to its content after creation — verified by maximize + programmatic MoveWindow; if you ever see a runaway, check your *test harness*, not the app — PowerShell `$h`/`$H` collide case-insensitively). Clicks map back via the inverse (`to-design`): `design = (offset − fit-ofs) / fit-s`. **Aspect-locked *window* dragging is NOT possible** — the backend's `WM_SIZING` size-override is a documented `@@ TBD` (`events.reds`) — hence letterboxing. The zero-alloc render loop is intact: the prefix is rebuilt only on resize, never per frame. Don't verify DPI/scale layout with self-captured screenshots. (memory `feedback_red_view_dpi`, `project_red3d`)
- **Draw can't take an inlined font OBJECT** — `font (fnt-obj)` (a composed value) silently halts the Draw dialect parser mid-block. Reference fonts by **bare word**: `font fnt-title`. (memory `feedback_red_draw`)
- **`does`/`func` do NOT auto-localise set-words** — the `build-*` Draw builders all assigned a shared global `out` and clobbered each other (chrome vanished). They're `function` (auto-local); `render`/`tick` stay `does` because they intentionally mutate globals (`pat-cache`, `last-row`, `framecount`).
- Reserved word: a param named `off` is the R/S logic literal `false` (→ use `ofs`). Pointer deref is `/value`, not `/1`. R/S modulo is `//`.
- **Master volume = SDL output gain** (`SDL_SetAudioStreamGain`), NOT `XMP_PLAYER_VOLUME`. libxmp's `xmp_channel_info.volume` is the POST-master final volume (`player.c`: `ci->volume = info_finalvol >> 4`), so driving the master through libxmp dims the VU meters with it. SDL gain leaves the decode + meters at unity and applies at pull time (instant, affects already-queued audio). Volume is 0–100 in the UI → gain `v/100.0`.
- **Pattern effect glyph must be bounds-safe**: MOD `fxt` is 0–15 but IT/S3M/XM effect codes exceed 15; an out-of-range Red `series/(idx)` path returns `none` (printed literally as "none"). Use the 36-char `FXCH` table + `fx-type` (`?` past 35).
- **Never guess font metrics** — estimated per-char widths caused border-touching text, and top-anchored text sat at cell BOTTOMS (a 15pt line box is ~25px in a 20px cell; digits ride the baseline, low in the box). Measure once at the first tick (View must be up) via `size-text` on a hidden scratch face: per-font advance `aw-* = sz/x / 8.0` AND line height `th-* = sz/y`. Derive every alignment from them: widths for centering/right-align/cell sizes, heights via `cen-y` (center the LINE BOX: `(cell-h - th) / 2`, negative is fine) for all vertical placement. `measure-fonts` also derives the time-cell/seek-groove geometry. For SYMBOL glyphs (the transport icons) even the line box lies — ink can ride low in the em box (↻ did) — so `measure-icon-inks` renders each glyph offscreen once, scans the pixels for the real ink top/bottom, and centring uses those. **Channel-number labels (`emit-chan-notes`) auto-shrink** from `fnt-lbl` (13 bold) to `fnt-chan` (11 bold, advance `aw-chan`) once a 2-digit number no longer clears the column (`(2*aw-lbl)+6 > colw`, ~44+ channels) — many-channel modules (e.g. the 47-ch Turrican II IT) overlapped otherwise; centring uses the measured advance, never a per-char guess.
- **FlodPro skin rules (user-set):** NO black text boxes in the chrome — value cells are sunken GRAY (`sunken-strip`) with dark `col-ink` data text; labels are WHITE (`col-white`); buttons white-enabled/dim-disabled. Dark panels are only for the pattern area + spectrum well.
- **Draw `shadow` does NOT render on this build** — `shadow <offset> <blur> <spread> <color>` PARSES (no reset form exists either) but produces zero pixels on the GDI+ backend (offscreen probe: 0 shadow pixels with and without a `scale` CTM). Emulate drop shadows by drawing the element twice: a translucent dark pass at +2x+2 first, then the real one on top (see `build-wordmark`). Beware false-positive "verification": sampling for dark pixels near glyphs can pick up bevel LINES — probe on a clean offscreen image, not the live UI. Gradient pens DO fill text glyphs (`pen linear ... text ...`) — probed offscreen.
- **Tintable glyphs need a MONOCHROME font.** The wordmark flourish (the cherry-red 🎶 before "Cherry") is drawn with **Segoe UI Symbol** so a gradient `pen` fills it like text; the **Segoe UI Emoji** version of the same codepoint renders in fixed full-colour and IGNORES the pen (probed offscreen). The monochrome shapes also differ from the colour emoji (Segoe UI Symbol's `1F3B5` is a one-flag eighth note, not the emoji's two-flag sixteenth; there is no tintable *single* two-flag note — `266C ♬` is the nearest, a beamed pair). `ink-bounds` (offscreen pixel-scan) centres the Segoe note's ink on the Consolas cap band, since the two fonts' line boxes don't align. The old runtime-loaded `cherry256.png` bitmap logo was dropped for this (file kept locally, gitignored).
- **Red tuple alpha = TRANSPARENCY (0 = opaque, 255 = clear)** — NOT opacity — in flat `fill-pen` colours AND gradient stops. Getting it backwards washed the VU bars white and silently blacked out the spectrum bars (only their caps, drawn after the overlay, stayed visible — gain was tuned for rounds against invisible bars). For "X% opaque" use `255 - (X*255/100)`. When shading misbehaves, don't iterate on app screenshots: probe offscreen — `img: draw 600x120 [known fills over a known bg]  save %t.png img` via red.exe, read pixels with PIL (one minute, definitive).
- **No `rate`/`on-time` timer — render from a MANUAL loop (close-race fix).** A `rate` timer dispatches the render `tick` as an EVENT; clicking the ✕ enters a Windows (and GTK) modal loop that keeps pumping those timer events *while the window tears down*, and that interleaving occasionally corrupts Red's per-event try-frame state (`make-event`'s `stack/mark-try-all` + `catch CATCH_ALL_EXCEPTIONS`, events.reds:622) → an escaped THROW = **"Runtime Error 95: no CATCH for THROW"** on close (rare on Windows, more frequent on GTK). A `closing?` latch + `if closing? [exit]` guard does NOT fix it: the racy ticks fire *before* `on-close` runs (during the button press), and `exit` is itself a return-throw. `PostMessage WM_CLOSE` can't reproduce it (it skips the modal loop) — only a real ✕ click does. **Fix = drive rendering yourself, no PERMANENT timer:** `view/no-wait win` then `while [win/state][ if timer-on? [canvas/rate: none timer-on?: no]  tick  do-events/no-wait  if win/state [wait 0:0:0.01] ]  pt-quit`. `tick` is called DIRECTLY (not as an interleaved event), the close is processed by `do-events`, and `win/state` going none ends the loop deterministically — on the close path there are no timer events to interleave with teardown ⇒ the race class is gone. `wait` paces ~60fps at low CPU (same granularity ceiling `rate` hit); busy-spin pumping `do-events` is the smoother-but-CPU-hungry alternative. **Live render during move/resize:** the OS modal move/resize loop BLOCKS the manual loop, so playback/animation would freeze for the drag. To keep it alive, arm a TRANSIENT `rate 60` timer for the drag's duration only — `on-resizing`/`on-moving` (fire only inside the size/move modal loop, which DOES pump `WM_TIMER`) set `canvas/rate: 60`; the manual loop disarms it the instant it regains control. The ✕-button modal loop fires NEITHER actor, so the timer stays off on the close path and the race can't return; `on-close` also kills it defensively. `on-resizing` must relay out (not just `on-resize`): EVT_SIZE is suppressed mid-drag while `win-state=1` (events.reds:1675). (memory `feedback_red_view_realtime`)
